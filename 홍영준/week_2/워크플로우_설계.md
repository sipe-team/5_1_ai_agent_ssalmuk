# Code QnA: 워크플로우 설계

## 전체 워크플로우

Slack 질문 접수부터 소스코드 기반 답변 응답까지의 End-to-End 자동화 흐름을 정의한다.

### Phase 1: 질문 접수 및 의도 분류

```
[Slack 메시지 수신] → [의도 분류 에이전트] → intent 판별
                                                │
                                                ├─ "question" → Phase 2로 진행
                                                ├─ "stop" → 진행 중 작업 중단
                                                └─ "other" → 기타 응답 처리
```

1. 사용자가 Slack에서 봇을 멘션하며 질문
2. 해당 스레드의 전체 컨텍스트를 포함하여 서버로 전달
3. 즉시 "답변 생성 중" 임시 댓글을 Slack 스레드에 추가
4. 의도 분류 에이전트가 메시지를 분석하여 intent 판별

### Phase 2: 화면 식별 및 저장소 매핑

```
[질문 텍스트] → [화면/키워드 추출 에이전트] → 화면명 + 키워드
                                                  │
                                                  ▼
              [화면 매핑 검색 에이전트] → 화면 매핑 DB 조회
                                                  │
                                      ├─ 매핑 발견 → 저장소 URL 획득
                                      └─ 매핑 없음 → 등록 안내 응답
```

1. 화면/키워드 추출 에이전트가 질문에서 화면명과 키워드를 추출
2. 추출된 정보로 화면 매핑 DB에서 유사도 기반 검색 수행
3. 매핑된 Git 저장소 URL을 획득
4. 화면 매핑이 없는 경우, 어드민 등록 링크를 안내하고 종료

### Phase 3: 소스코드 분석 및 답변 생성

```
[저장소 URL + 질문] → [코드 분석 에이전트 (LangGraph)]
                                    │
                                    ▼
                          ┌─── ReAct 루프 ───┐
                          │                   │
                          │  [LLM 추론]       │
                          │      │            │
                          │      ▼            │
                          │  [Git Proxy API]   │
                          │  - 프로젝트 검색  │
                          │  - 파일 트리 탐색 │
                          │  - 파일 읽기      │
                          │  - 코드 검색      │
                          │      │            │
                          │      ▼            │
                          │  [결과 관찰]      │
                          │      │            │
                          │   충분? ──No──→   │
                          │      │            │
                          │     Yes           │
                          └──────┼────────────┘
                                 ▼
                          [최종 답변 생성]
```

1. 코드 분석 에이전트가 LangGraph ReAct 패턴으로 실행
2. Git Proxy API를 통해 저장소의 파일 구조 탐색 → 관련 파일 식별 → 소스코드 읽기
3. 각 단계마다 `onStep` 콜백으로 Slack "답변 생성 중" 메시지를 업데이트 (진행 상태 표시)
4. 충분한 정보를 수집하면 최종 답변 생성

### Phase 4: 응답 전달

```
[최종 답변] → [Slack 스레드에 새 댓글 추가] → 사용자 알림
           → [질의응답 이력 DB 저장]         → 이력 관리
```

1. 최종 답변을 Slack 스레드에 새 댓글로 추가 (알림 발생)
2. 질의응답 이력을 DB에 저장
3. 기존 "답변 생성 중" 임시 메시지는 최종 상태로 업데이트

## 예외 처리 흐름

### 메시지 수정 시

```
[유저 메시지 수정 감지] → [기존 작업 AbortSignal 발동]
                        → [수정된 질문으로 Phase 1부터 재시작]
```

- 사용자가 질문을 수정하면 기존 진행 중인 분석을 즉시 중단
- 수정된 질문으로 처음부터 다시 시작

### 동일 스레드 추가 질문 시

```
[스레드 내 추가 멘션 감지] → [기존 작업 완료 대기 또는 큐잉]
                           → [새 질문으로 Phase 1 시작]
```

- 진행 중인 답변이 있을 때 추가 질문이 들어오면 순차 처리

### 에러 발생 시

| 에러 유형 | 처리 방식 |
| --- | --- |
| 화면 인식 실패 | "질문에서 화면을 식별하지 못했습니다" 안내 |
| 화면 매핑 미등록 | 어드민 등록 링크 안내 |
| 저장소 URL 미등록 | 화면 매핑에 저장소 등록 요청 안내 |
| Agent 실행 실패 | "답변 생성에 실패했습니다" 에러 메시지 |
| 작업 중단 | "작업이 중단되었습니다" 안내 |
| LLM 서비스 장애 | 헬스체크 실패 알림 |

## UX 고려사항

| 항목 | 설계 |
| --- | --- |
| 알림 최소화 | 진행 중에는 기존 댓글을 수정하고, 최종 답변 시에만 새 댓글 추가 |
| 진행 상태 표시 | "답변 생성 중" 메시지에 현재 분석 단계를 업데이트 |
| 중단 지원 | "그만해" 등 자연어로 진행 중 작업 중단 가능 |
| 메시지 수정 대응 | 수정 감지 시 자동으로 기존 작업 취소 후 재시작 |
