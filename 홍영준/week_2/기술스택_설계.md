# Code QnA: 기술 스택 설계

## 기술 스택 개요

| 영역 | 기술 | 비고 |
| --- | --- | --- |
| 언어 | TypeScript | NestJS 기반 서버 |
| LLM | Claude Sonnet/Opus | 의도 분류, 키워드 추출, 코드 분석 |
| 에이전트 프레임워크 | LangGraph.js | ReAct 패턴 기반 단일 에이전트 오케스트레이션 |
| 데이터 저장소 | PostgreSQL | 화면-저장소 매핑, 질의응답 이력 |
| Git 저장소 연동 | Git Proxy API (내부 서버 HTTP) | 소스코드 탐색 및 파일 읽기 |
| 메시지 연동 | Slack API | 질문 수신, 답변 스레드 응답 |
| API 서버 | NestJS | REST API + 모듈 구조 |
| 컨테이너 | Docker | 배포 환경 |

## 에이전트별 기술 상세

### 의도 분류 에이전트

- **LLM**: Claude Sonnet/Opus
- **프롬프트**: 프롬프트 기반 intent 분류
- **출력 형식**: JSON (`{ intent: "question" | "stop" | "other" }`)

### 화면/키워드 추출 에이전트

- **LLM**: Claude Sonnet/Opus
- **프롬프트**: Few-shot 예시 기반 추출
- **출력 형식**: JSON (`{ screen: string, keywords: string[] }`)

### 화면 매핑 검색 에이전트

- **데이터 소스**: PostgreSQL 화면 매핑 테이블
- **검색 방식**: 화면명/키워드 유사도 매칭
- **확장 계획**: pgvector 기반 벡터 유사도 검색 (RAG 전환 시)

### 코드 분석 에이전트

- **LLM**: Claude Sonnet/Opus
- **프레임워크**: LangGraph.js (ReAct 루프)
- **Tool**: Git Proxy API (프로젝트 검색, 파일 탐색, 파일 읽기, 코드 검색, 브랜치 조회)
- **출력**: Markdown 형식 답변

## LLM 호출 구조

OpenAI API 호환 형식의 LLM Gateway를 통해 LLM을 호출한다. 표준 인터페이스를 사용하므로 모델 교체가 용이하다.

```
[Agent/Service] → [LlmService] → [LLM Gateway] → [LLM]
```

### LlmService 메서드

| 메서드 | 용도 | 사용처 |
| --- | --- | --- |
| `chat()` | 기본 LLM 호출 (Tool 포함) | 코드 분석 에이전트 메인 루프 |
| `simpleChat()` | 간단한 텍스트 응답 | 일반 대화 처리 |
| `chatForJson()` | JSON 형식 응답 요청 | 의도 분류, 화면/키워드 추출 |
| `healthCheck()` | 서비스 상태 확인 | 헬스체크 API |

## Git Proxy API 연동

### 개요

배포 환경에서 Git 저장소 접근을 위해 내부 서버의 Git Proxy API를 HTTP로 호출한다.

### 제공 Tool 목록

| Tool | 기능 |
| --- | --- |
| 프로젝트 검색 | 저장소 이름으로 프로젝트 검색 |
| 파일 내용 조회 | 특정 파일의 소스코드 읽기 |
| 디렉토리 트리 조회 | 저장소의 파일/폴더 구조 탐색 |
| 브랜치 목록 조회 | 프로젝트의 브랜치 목록 확인 |
| 코드 검색 | 프로젝트 내 키워드 기반 코드 검색 |

### 연동 구조

```
[코드 분석 에이전트] → [ToolRegistry] → [Git Proxy API Tool] → [내부 서버 HTTP]
                                                                      │
                                                                ├─ 프로젝트 검색
                                                                ├─ 파일 트리 탐색
                                                                ├─ 파일 내용 읽기
                                                                ├─ 브랜치 조회
                                                                └─ 코드 검색
```

## 고려 사항

- **비용 최적화**: 단순 분류/추출 작업은 경량 모델, 코드 분석은 고성능 모델 사용 가능하도록 모델 분리 설계
- **확장성 (RAG)**: 초기 PoC는 Git Proxy API 기반 실시간 탐색, 이후 RAG(벡터 검색) 기반으로 성능 개선 가능한 구조
- **확장성 (Branch)**: 초기에는 main 브랜치 기준, 이후 최근 배포 브랜치 등 동적 브랜치 탐색 지원
- **프롬프트 관리**: 프롬프트를 코드와 분리하여 유지보수 용이성 확보
- **모듈 재사용**: Agent 모듈을 공통 인프라로 설계하여 Code QnA 외 다른 AI 기능에서도 재사용
- **장애 대응**: LLM 서비스 장애 시 graceful 실패 + 헬스체크 API 제공
