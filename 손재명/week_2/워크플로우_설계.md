# Week 2: 워크플로우 설계

## 전체 워크플로우

장애 접수부터 포스트모템까지의 End-to-End 자동화 흐름을 정의한다.

### Phase 1: 장애 접수 및 자동 분류

```
[장애 접수] → [장애 분류 에이전트] → 카테고리 + 심각도 판단
                                        │
                                        ├─ Critical → 즉시 에스컬레이션
                                        ├─ Major → 30분 내 대응
                                        └─ Minor → 일반 큐 등록
```

1. 사용자가 장애를 접수 (Slack/웹 폼/API)
2. 장애 분류 에이전트가 내용을 분석하여 카테고리 및 심각도 자동 판단
3. 심각도에 따라 대응 우선순위 결정

### Phase 2: 유사 장애 검색 및 담당자 배정

```
[분류 결과] → [유사 장애 검색 에이전트] → 유사 사례 + 1차 조치 가이드
          → [담당자 매칭 에이전트]   → 담당자 자동 배정 + 알림
```

1. 분류 결과를 기반으로 유사 장애 검색과 담당자 매칭을 **병렬** 실행
2. 유사 장애 검색 결과와 1차 조치 가이드를 담당자에게 함께 전달
3. 담당자에게 Slack/이메일 알림 발송

### Phase 3: 대응 및 진단

```
[담당자 대응 시작] → [런북 에이전트] → 단계별 조치 플레이북 제공
                  → [심층 진단 에이전트] → 대화형 원인 분석
```

1. 런북 에이전트가 해당 장애 유형에 맞는 조치 플레이북 제공
2. 필요 시 심층 진단 에이전트와 대화형으로 원인 분석 수행
3. 각 조치 단계를 타임라인에 자동 기록

### Phase 4: 종결 및 보고

```
[장애 해결] → [포스트모템 에이전트] → 보고서 초안 생성
           → [장애 이력 DB 저장]   → RAG 벡터 스토어 업데이트
```

1. 장애 해결 후 포스트모템 에이전트가 보고서 초안 자동 생성
2. 해결 이력을 장애 DB에 저장
3. 벡터 스토어에 임베딩 업데이트 (향후 유사 장애 검색에 활용)

## SLA 모니터링

각 Phase별 제한 시간을 설정하고, 위반 시 자동 에스컬레이션:

| Phase | 심각도 Critical | 심각도 Major | 심각도 Minor |
| --- | --- | --- | --- |
| Phase 1 (분류) | 5분 | 10분 | 30분 |
| Phase 2 (배정) | 10분 | 20분 | 1시간 |
| Phase 3 (대응) | 1시간 | 4시간 | 24시간 |
| Phase 4 (종결) | 24시간 | 48시간 | 1주 |
