## [1순위] QA 버그 리포트 → 코드 연관 분석 + GitHub Issue 자동 생성

### 대상 업무

**QA/베타 사용자 버그 리포트를 GitHub Issue로 전환하는 전체 과정**

### 현재 문제

버그 리포트가 접수되면 아래 과정을 **모두 수동으로** 처리해야 함. 솔로 개발자 구조 특성상 이 병목이 고스란히 개발 지연으로 이어짐.

1. 리포트 내용을 읽고 어느 레이어(프론트/API/AI 파이프라인)에서 발생한 문제인지 판단
2. 관련 소스 파일·함수·API 엔드포인트를 직접 코드에서 탐색
3. 재현 조건, 영향 범위, 우선순위를 직접 판단하여 GitHub Issue 수동 작성
4. 관련 컴포넌트 레이블·마일스톤·담당자 태그 수동 설정

### 핵심 Pain Point

- 버그 하나당 코드 탐색 + Issue 작성에 **평균 20~40분** 소요
- 사용자 언어("사진이 안 나와요")와 코드 구조 사이의 **도메인 간격**을 매번 수동으로 메워야 함
- 베타 오픈 직후 리포트가 몰리면 처리 대기열이 쌓이고 **개발 컨텍스트 스위칭 비용**이 급증
- 리포트가 누락되거나 낮은 품질의 Issue가 생성되어 **나중에 다시 분석**해야 하는 경우 발생

### 반복 빈도

베타 오픈 이후 **주 5~10건** 이상 예상 (이벤트 트래픽 증가에 비례)

---

### AI로 해결 가능한 포인트

#### 1. 버그 리포트 자동 파싱 및 의도 분류

LLM이 자연어 버그 리포트를 읽고 **기능 영역**(사진 업로드 / 다운로드 / AI 매칭 / 결제 등)과 **심각도**(Critical / Major / Minor)를 자동 분류

#### 2. 관련 코드 자동 탐색 (RAG + 코드베이스 임베딩)

코드베이스를 벡터 임베딩으로 인덱싱하여, 버그 리포트와 유사도 높은 **소스 파일·함수·API 엔드포인트·프론트 컴포넌트**를 자동으로 검색 및 연결

> 예) "사진 다운로드가 안 돼요"
> → `/api/photos/download` 엔드포인트, CloudFront presigned URL 만료 로직, 프론트 `DownloadButton` 컴포넌트 자동 매핑

#### 3. 유사 과거 이슈 검색

기존 GitHub Issues를 임베딩 검색하여 **중복 리포트 감지** 및 유사 이슈 링크 자동 첨부 — 재발 버그의 빠른 컨텍스트 복원 가능

#### 4. GitHub Issue 초안 자동 생성

분석 결과를 기반으로 아래 항목이 채워진 **구조화된 Issue 초안**을 자동 생성:
- 제목 / 재현 조건 / 영향 범위
- 연관 코드 위치 (파일명 + 라인 범위)
- 재현 가능한 시나리오 추정
- 레이블·마일스톤·담당자 자동 태그

#### 5. 카카오톡 버그봇 연동 (입력 채널)

카카오톡으로 들어오는 사용자 버그 리포트를 봇이 수신 → 자동 분석 파이프라인 트리거 → Slack/GitHub으로 결과 전달

---

### 기대 효과

| 항목 | AS-IS | TO-BE |
| --- | --- | --- |
| 버그 리포트 → Issue 생성 시간 | 수동 20~40분 | 자동 분석 후 검토만 2~5분 |
| 코드 탐색 | 개발자가 직접 grep/탐색 | RAG로 관련 파일 자동 제시 |
| Issue 품질 | 작성자 역량에 따라 들쭉날쭉 | 구조화된 표준 템플릿 자동 생성 |
| 중복 리포트 처리 | 중복 인지 못하고 재작업 | 유사 이슈 자동 감지 및 링크 |
| 베타 트래픽 대응 | 리포트 몰리면 처리 지연 | 수신 즉시 병렬 자동 처리 |

## [2순위] AI 파이프라인 자동 진단 리포트 봇

### 대상 업무

**이벤트별 AI 추론 파이프라인 결과 검수 및 셀피 미스매칭 원인 분석**

### 현재 문제

AI 추론 파이프라인 자체는 자동화되어 있지만, **결과 검증은 전적으로 수동**. 이벤트가 끝나도 결과가 정상인지 확인하려면 직접 S3에서 데이터를 꺼내 로그를 뒤져야 하며, 셀피 미스매칭이 발생해도 어느 단계에서 왜 틀렸는지 파악하는 것 자체에 시간이 걸림. 문제를 늦게 발견하면 고객 클레임으로 이어짐.

### 핵심 Pain Point

- 이벤트 완료 후 결과 이상 여부를 확인하는 데 **이벤트당 30분 이상** 소요
- 미스매칭 원인 분석은 각 단계 로그를 수동으로 대조해야 하므로 **원인 파악에 수 시간** 걸리기도 함
- 문제를 뒤늦게 발견하면 재처리 비용 + **고객 클레임 리스크** 발생
- 이벤트 수가 늘어날수록 수동 검수는 **선형적으로 불가능**해짐

### 반복 빈도

**이벤트가 처리될 때마다** (이벤트 수 증가에 비례하여 누적)

---

### AI로 해결 가능한 포인트

#### 1. 이벤트 완료 시 자동 결과 리포트 생성

파이프라인 종료 트리거에 맞춰 매 이벤트마다 결과 요약 리포트를 자동 생성. 클러스터 수, confidence score 분포, 매칭률, 미감지 사진 비율 등을 집계하여 한눈에 파악 가능한 형태로 Slack/이메일 발송.

> 예) "이벤트 #142 완료 — 총 823장, 클러스터 47개, 평균 confidence 0.91, 미매칭 사진 12장 (1.5%) ⚠️ 임계값 초과"

#### 2. 이상 감지 및 자동 알림

주요 지표(미매칭률, 평균 confidence, 처리 소요 시간)에 임계값을 설정하고, 초과 시 **즉시 알림** 발송 — 문제를 사전에 감지하여 클레임 예방

#### 3. 셀피 미스매칭 발생 시 원인 분석 추가 리포트

매칭 실패 케이스를 감지하면 해당 셀피와 후보군 사진의 메타데이터(해상도, 조명 추정, 얼굴 각도, 등록 시점 등)를 비교 분석하여 원인 후보를 자동으로 제시.

> 예) "셀피 #user_039 미스매칭 — 원인 후보: 셀피 해상도 저하(280px), 이벤트 사진 대비 조명 편차 큼, 유사도 top-3 모두 0.6 미만"

#### 4. 이벤트 간 트렌드 분석

누적 이벤트 데이터를 기반으로 **시계열 패턴**(특정 시간대 미스매칭 증가, 특정 이벤트 유형 이상 발생 등)을 자동 탐지하여 인프라·모델 개선 우선순위 도출

---

### 기대 효과

| 항목 | AS-IS | TO-BE |
| --- | --- | --- |
| 이벤트 결과 확인 | S3·로그 수동 조회, 30분+ | 파이프라인 완료 시 자동 리포트 수신 |
| 이상 감지 시점 | 고객 클레임 접수 후 인지 | 이벤트 완료 직후 자동 알림 |
| 미스매칭 원인 분석 | 로그 수동 대조, 수 시간 소요 | 메타데이터 자동 비교, 원인 후보 즉시 제시 |
| 이벤트 증가 대응 | 검수 인력/시간 선형 증가 | 이벤트 수와 무관하게 자동 처리 |
