# Week 2: 워크플로우 설계

## 실행 구조

**함수형 단발 실행** — 버그 리포트 1건 등록 시 파이프라인이 1번만 실행되고 종료된다. 상태 유지 없음, 재시도 없음, 캐싱 인프라 없음. 단순하게 유지한다.

---

## 멀티 레포 구성


| 레포                  | 담당 영역                 |
| ------------------- | --------------------- |
| `aipics-api-server` | REST API, 비즈니스 로직, DB |
| `aipics-web`        | 프론트엔드 (Next.js)       |
| `inference-server`  | AI 추론 파이프라인           |


**멀티 레포 원칙:**

- 동일 버그로 2개 이상 레포 수정이 필요한 경우 동일 타이틀로 각 레포에 Issue 개별 등록
- 각 Issue 본문 하단에 연관 레포 Issue 크로스 링크 자동 첨부
- `inference` 레포는 Suggestion Agent 수정 제안 대상에서 제외

## 전체 워크플로우

### Phase 1: 버그 접수 및 자동 분류

```
[어드민 게시판 버그 등록]
  → POST /api/admin/bugs
  → Classifier Agent (Haiku) 즉시 트리거
  → severity + affected_layer 판단
      │
      ├─ unknown + low  → "수동 분류 필요" 반환, 파이프라인 종료  ← 비용: Haiku 1회
      └─ 그 외          → Phase 2 진행

```

1. 어드민 게시판에서 버그 등록
2. Classifier Agent가 기능 영역·심각도·영향 레이어·대상 레포 자동 판단
3. 분류 불가(`unknown + low`) 시 즉시 종료 — 이후 LLM 호출 없음

---

### Phase 2: 중복 감지 + 코드 탐색 (병렬 실행)

```
[Classifier 결과]
  ├──▶ [Issue RAG Agent]  pgvector 유사도 검색 (LLM 없음)
  └──▶ [Code RAG Agent]   pgvector 검색 → Haiku 리랭킹
           │                      │
           ├─ 유사도 0.95↑ 중복      ├─ 연관 파일 0건
           │  → 기존 이슈 링크만      │  → Suggestion Agent 스킵 플래그
           │    등록 후 종료         │
           └──────────┬───────────┘
                      ▼
             [Phase 3 진행]

```

1. Issue RAG + Code RAG 병렬 실행 (LangGraph parallel node)
2. **중복 감지 시**: Code RAG 결과 무시, Issue Generator에 기존 이슈 링크만 전달 후 종료 → 비용: Haiku 합산 ~2회
3. **연관 파일 0건**: Suggestion Agent 호출 플래그 false로 설정

---

### Phase 3: GitHub Issue 생성

```
[분석 결과 종합 — 압축된 형태]
  → Issue Generator Agent (Sonnet 4.5)
  → affected_repos 수만큼 Issue 등록
      ├─ repos = 1개 → 해당 레포 Issue 1건
      └─ repos = N개 → 동일 타이틀로 각 레포 Issue N건
                       (본문에 크로스 링크 자동 첨부)
  → 어드민 status: "issue_created"
  → Slack 알림 (레포별 Issue URL)

```

1. 각 에이전트의 압축 출력만 입력으로 사용 — 원문·파일 원본 미포함
2. Issue 템플릿은 Prompt Caching prefix로 고정
3. 레이블·담당자 자동 태그 포함

---

### Phase 4: 수정 제안 코멘트 등록 (조건부)

```
[어드민 검토 후 승인]
  → severity 확인
      ├─ medium / low → Suggestion Agent 호출 없음, 종료        ← 비용 0
      └─ critical / high → Suggestion Agent (Sonnet 4.5) 실행
           │
           ├─ affected_layer = inference → 제안 생략, 종료
           └─ 그 외 → 함수 블록 Top-3 기반 diff 제안 생성
                      → 레포별 Issue에 코멘트 등록
                      → 어드민 status: "suggestion_ready"
                      → Slack 알림

extended thinking:
  - critical만 활성화
  - high 이하는 일반 모드

```

1. severity = medium/low이면 Suggestion Agent 호출 자체 없음 → Sonnet 비용 0
2. 함수 블록만 컨텍스트에 포함, 파일 전체 전달 금지
3. 실제 브랜치·커밋·PR 없음, 코멘트 제안만 등록

---

### Phase 5: 이력 저장

```
[PR 머지 완료] (GitHub webhook)
  → 버그 DB status: "resolved"
  → 해결된 Issue → Issue RAG 벡터스토어 증분 추가

```

## 시나리오별 LLM 호출 비용 비교


| 시나리오                | 호출 에이전트                            | 모델                   | 비고                  |
| ------------------- | ---------------------------------- | -------------------- | ------------------- |
| 분류 불가 (unknown+low) | Classifier                         | Haiku ×1             | 조기 종료               |
| 중복 버그 감지            | Classifier + RAG 리랭킹               | Haiku ×2             | 조기 종료               |
| 신규 low/medium 버그    | Classifier + RAG + Issue Generator | Haiku ×2 + Sonnet ×1 | Suggestion Agent 없음 |
| 신규 high 버그          | 전체                                 | Haiku ×2 + Sonnet ×2 | 일반 모드               |
| 신규 critical 버그      | 전체                                 | Haiku ×2 + Sonnet ×2 | extended thinking   |

## SLA 기준


| Phase                | Critical    | High        | Medium / Low |
| -------------------- | ----------- | ----------- | ------------ |
| Phase 1 — 분류 완료      | 즉시 (자동)     | 즉시 (자동)     | 즉시 (자동)      |
| Phase 2~3 — Issue 생성 | 5분 이내 (자동)  | 5분 이내 (자동)  | 수동 트리거 가능    |
| Phase 4 — 수정 제안      | 어드민 승인 후 즉시 | 어드민 승인 후 즉시 | 해당 없음        |

## API 설계

```
POST /api/admin/bugs                    → 버그 등록 + Phase 1~3 자동 실행
GET  /api/admin/bugs                    → 목록 조회
POST /api/admin/bugs/:id/suggest        → Phase 4 실행 (Suggestion Agent 수동 트리거)
GET  /api/admin/bugs/:id/status         → 진행 상태 SSE 스트림

```
