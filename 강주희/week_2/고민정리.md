## 1. Suggestion Agent — 코드 직접 수정 vs 제안만

**고민** 에이전트가 코드 수정까지 하게 해도 괜찮을까? 제안을 할 정도면 어차피 코드 수정까지 할 수 있는 거 아닌가?

**논의된 리스크**

- 여러 레포에 걸친 수정 시 인터페이스 불일치 리스크
- inference 레포는 AI 파이프라인 로직 특성상 사이드이펙트 예측 어려움
- DB 스키마 변경이 수반되는 경우
- 솔로 개발자 구조에서 "내가 짜지 않은 코드"가 쌓이면 6개월 후 유지보수 컨텍스트 부재 문제

**결론** 코드 직접 수정 없이 Issue 코멘트에 diff 형식 제안만 등록. 실제 구현과 PR은 개발자가 직접 수행. inference 레포는 제안도 "파이프라인 검토 필요" 수준으로 제한.

**운영 원칙**

- 브랜치 생성·커밋·PR 없음
- extended thinking은 severity = critical/high일 때만 활성화

**제안 방식으로 바꾼 진짜 이유 — 토큰 절약이 아니다**

제안이든 직접 수정이든 LLM이 하는 작업은 동일하다. 함수 블록 읽기 → 버그 원인 분석 → 수정 코드 생성. diff를 코멘트에 쓰나 파일에 커밋하나 LLM 토큰 소비는 똑같다. 차이가 나는 건 GitHub API 호출(브랜치 생성·커밋·PR)인데 이건 토큰이 아니라 GitHub API 비용이고 사실상 무료 수준.

실제 이유는 두 가지였다:

- **설계 단순화**: 브랜치 생성·커밋·PR 로직을 구현하지 않아도 됨
- **Human-in-the-loop 유지**: AI가 생성한 코드가 리뷰 없이 레포에 직접 반영되는 것을 방지

처음 문서화할 때 "토큰 절약" 맥락으로 묶인 것은 잘못된 분류였음. **근거: 설계 단순화 + Human-in-the-loop** 로 정정.

## 2. 멀티 레포 구성 (apiserver / web / inference)

**고민** 레포가 3개로 분리되어 있어서 두 곳을 동시에 수정해야 하는 버그면 Issue를 각각 등록해야 하는데, 이걸 어떻게 처리할까?

**결론**

- Classifier Agent 출력에 `affected_repos[]` 필드 추가
- `affected_repos`가 2개 이상이면 동일 타이틀로 각 레포에 Issue 개별 등록
- 각 Issue 본문 하단에 연관 레포 크로스 링크 자동 첨부

```
예) apiserver#142 본문 하단
Related: web#89 (frontend fix required)

```

## 3. Redis 캐싱 — 필요한가

**고민** 토큰 절약을 위해 Redis 캐싱을 설계에 포함했는데, 실제로 필요할까?

**결론** 불필요. 전체 워크플로우가 버그 리포트 1건 등록 시 1번만 실행되는 함수형 단발 실행 구조이므로 캐싱 자체가 의미 없음. 설계에서 제거.

**왜 Redis가 설계에 들어갔었나** "토큰 절약"이라는 키워드에 반응해서 캐싱도 절약 방법이라는 이유로 맥락 없이 붙인 결과였음. 근거로 썼던 로직은 "같은 버그가 여러 번 분석 요청될 수 있으니 중복 LLM 호출을 막자"였는데, 버그 리포트 1건 = 파이프라인 1회 실행 구조에서는 애초에 성립하지 않는 가정이었음. **설계 일관성보다 항목 채우기가 앞선 케이스** — 워크플로우 실행 구조를 먼저 확정한 뒤 필요한 컴포넌트만 추가하는 순서로 접근했어야 했음.

## 4. 토큰 절약 — 진짜 낭비 포인트는 어디인가

**고민** 현재 설계가 토큰을 많이 낭비하나?

**실제 낭비 포인트** Redis나 캐싱이 아니라 Suggestion Agent가 핵심. 연관 파일 원본을 컨텍스트에 넣으면 파일 하나에 수백~수천 토큰, Top-3 파일이면 5k~15k 토큰 초과 가능. extended thinking까지 켜면 input/output 토큰 대폭 증가.

**채택한 절약 전략**

- 파일 전체 대신 함수 블록만 컨텍스트에 포함
- extended thinking은 critical/high 버그만 활성화
- 제안 대상 파일 수 Top-2~3으로 제한
- Issue RAG는 LLM 호출 없이 pgvector 검색만으로 처리
- Code RAG 리랭킹은 Haiku로 처리
- 유사도 0.95 이상 중복 감지 시 이후 파이프라인 조기 종료

## 추가로 고려할 포인트

### A. 버그 리포트 품질 편차 대응

사용자가 입력하는 버그 리포트는 품질 편차가 큼. "안 돼요" 한 줄짜리 리포트가 들어오면 Classifier가 `affected_layer = unknown`으로 분류하고 이후 파이프라인이 의미 없는 결과를 낼 수 있음.

**고려할 방향**

- 어드민 게시판 입력 폼에 재현 조건·기기 환경·스크린샷을 필드로 강제
- Classifier가 리포트 품질 점수를 함께 출력하여 일정 점수 이하면 "추가 정보 요청" 상태로 보류

---

### B. 코드베이스 인덱스 커버리지 한계

Tree-sitter AST 파싱은 정적 분석이므로 런타임 동작(동적 라우팅, 조건부 import, 환경변수 기반 분기)은 인덱스에 잡히지 않음. 이 경우 Code RAG가 연관 파일을 아예 못 찾을 수 있음.

**고려할 방향**

- 인덱스에 없는 경우를 위한 fallback: 파일명/디렉토리 구조 기반 키워드 매칭 레이어 추가
- 런타임 로그 기반 보완 (API 요청 로그에서 실제 호출된 엔드포인트 추적)

---

### C. Suggestion Agent 제안 품질 검증

LLM이 생성한 diff 제안이 실제로 올바른지 확인할 수단이 없음. 잘못된 제안을 개발자가 그대로 적용할 리스크 존재.

**고려할 방향**

- 제안 코멘트에 항상 "검증 필요 항목" 체크리스트 포함 (테스트 케이스, 엣지케이스 등)
- 향후 CI 연동 시 제안 코드를 샌드박스에서 lint/테스트 실행 후 결과 첨부

---

### D. 에이전트 실행 실패 처리

LLM API 오류, GitHub API rate limit, pgvector 쿼리 타임아웃 등 중간 단계 실패 시 처리 전략 필요.

**고려할 방향**

- 각 Phase별 실패 시 어드민 게시판에 실패 단계와 이유 표시
- Phase 1(분류) 실패 시에만 전체 재시도, 이후 Phase는 해당 Phase부터 재시작 가능하도록 상태 저장
- GitHub API rate limit 대비 지수 백오프 재시도

---

### E. 장기적: 버그 패턴 누적 분석

버그 이력이 쌓이면 "어떤 기능 영역에서 버그가 자주 발생하는가"를 분석할 수 있음. 현재 설계는 건별 처리에 집중되어 있어 이 데이터를 활용하지 못함.

**고려할 방향**

- 버그 DB에 `feature_area + severity + resolved_at` 누적
- 월간 자동 리포트: 재발 버그 Top-5, 평균 해결 소요 시간, 레포별 버그 분포
- 2순위 과제(AI 파이프라인 자동 진단 리포트 봇)와 연계 가능
